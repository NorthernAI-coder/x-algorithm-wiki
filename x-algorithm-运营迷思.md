## 先花 30 秒,看懂这套算法的骨架

一条帖子要进你的 For You,走两步:

- **召回**:从全网海量帖子里粗筛出几百条候选,求快。
- **排序**:用一个 Grok 改造来的 transformer,给每条候选精细打分。

排序这步是重点。这个模型**不预测一个笼统的"喜欢度"**,而是同时预测大约 **22 种具体行为的概率**——你会不会点赞、会不会回复、会不会转发、会不会点开、会不会停留……也包括会不会点"不感兴趣"、会不会拉黑、会不会举报。

然后,`RankingScorer` 把这 22 个概率各乘一个权重,加起来,就是这条帖子对你的最终分(`home-mixer/scorers/ranking_scorer.rs:125-170`)。

骨架就这么简单。但有一个事实,是整篇文章的地基,也是最该被记住的一句:

> **这套打分系统,几乎没有人工规则。**

README 第一段的原话是:他们删除了系统里**每一个**手工特征和**大部分**启发式规则(`README.md:55`);"无手工特征"还被列为 Key Design Decision 的第一条(`README.md:324-325`)。

记住这句。下面六个迷思,有一半是它直接打穿的。

---

## 迷思一:「多发帖 = 多曝光」

**流行说法**:号要勤更新,一天多发几条,曝光自然摊大。

**翻开源码**:`RankingScorer` 打完加权分之后,有专门的一步叫**作者多样性衰减**(`ranking_scorer.rs:190-217`)。

它的逻辑是:把候选按分数排好后,给每个作者记一个出现计数 `position`。同一个作者第 N 次出现的那条帖子,分数要乘上一个衰减系数:

```
multiplier = (1 - floor) × decay^position + floor
```

`decay` 小于 1(`ranking_scorer.rs:186-188`)。所以同一作者的第 0 条乘 1.0(不打折),第 1 条开始打折,第 2 条、第 3 条……越往后乘数越小,一直衰减到地板值 `floor` 为止。

**所以**:在同一次信息流计算里,你的内容中第一条机会最大,后面每一条,都在被你自己的前一条往下压。

刷屏不是多曝光,是**自我稀释**。

(一个诚实的边界:这是"单次信息流计算之内"的衰减,源码里看不到账号级的永久惩罚。但对"今天多发几条就能多占位置"这个想法,方向已经很明确——它会反噬。)

---

## 迷思二:「互动量越高越好」

**流行说法**:想尽办法冲互动数据,点赞评论转发越多,权重越高。

**翻开源码**:最终分是 22 种**预测行为概率**的加权和。关键在于,这 22 个里有 **5 个是负向行为**:

- `not_interested`——用户点了"不感兴趣"
- `block_author`——拉黑你
- `mute_author`——静音你
- `report`——举报你
- `not_dwelled`——刷到你的帖子,**没停留就划走了**

这 5 个的权重是**负数**,在代码里汇成一个 `negative_sum = -(...)`(`ranking_scorer.rs:83`)。README 自己也写明:负向行为"have negative weights, pushing down content the user would likely dislike"(`README.md:292`)。

**所以**:算法算的从来不是"互动总量",而是 **"正向行为概率 − 负向行为概率"** 的加权结果。

靠标题党、靠骗点击换来的那点互动,如果同时招来一大片"划走""不感兴趣""举报",这条帖子的加权分会被**直接拉低,甚至变成负数**。

这里有一个非常反直觉、但写死在源码里的点:

> `not_dwelled`——"用户划走、没停留"——被**显式建模成一个扣分项**。

也就是说:**"没人理你"不是最差的结果;"很多人划走、举报你"才是。** 一条无人问津的帖子,负向信号也低;一条骗来曝光却让人反感的帖子,是净亏的。

---

## 迷思三:「做泛内容就能轻松破圈」

**流行说法**:内容够好,算法自然把你推给全网陌生人。

**翻开源码**:候选分两种——**站内**(in-network,你关注的人发的)和**站外**(out-of-network,你没关注的)。

`RankingScorer` 的最后一步,对这两种候选区别对待(`ranking_scorer.rs:272-275`):

```
站外候选:最终分 = 多样性调整分 × OON 系数
站内候选:最终分 = 多样性调整分        (不乘)
```

这个 OON 系数(`effective_oon`)取自配置参数(`ranking_scorer.rs:220-239`)。

**所以**:系统在结构上,对站外内容设了一道**折扣闸**。你的帖子要触达没关注你的人,是带着一个**小于 1 的乘数**,去和别人关注流里的内容竞争。

破圈不是不可能,但你得明显更强,才能追平这个折扣。"轻松破圈"是个错觉。

(边界:这个系数的**具体数值**是线上参数,不在开源仓库里。我能确定的是"机制方向是降权",不能告诉你"降到几折"——任何告诉你确切数字的解读,都是在编。)

## 迷思四:「套爆款模板、加特定关键词,能骗算法」

**流行说法**:存在一套"算法喜欢的"格式、关键词、结构,套上去就涨。

**翻开源码**:回到那条地基——README 明说,打分侧删掉了**每一个**手工特征(`README.md:55, 324-325`)。

排序模型靠的是 Grok transformer **直接读用户的行为序列**来学相关性。打分模型里**根本没有**"这条帖子带没带某个关键词""是不是某种排版格式"这一类人工特征。

关键词在整套系统里,**唯一的角色**是反过来的:用户自己设置的**屏蔽词**(`MutedKeywordFilter`)——命中就把帖子剔除,纯负向。它不是给创作者堆词用的加分位。

**所以**:不存在"算法喜欢的格式"这种东西可供你套用,因为打分侧压根没有识别格式、识别关键词的特征。

你能"骗"的不是算法,是**真实用户的行为**——模型预测的是"像这个用户的人,看到这类内容,会不会产生正向行为"。模板能不能涨,取决于真人买不买账,不取决于一条不存在的规则。

(一个必须说清楚的精确点:README 删的是**打分/排序侧**的手工特征。系统的**过滤侧**仍然是明确的规则——共 17 个过滤器,管帖龄、屏蔽词、拉黑关系、已看去重等等。准确的说法是:**打分靠学习,过滤靠规则**。把这两件事混为一谈的解读,基本可以判定没读懂这套系统。)

## 迷思五:「我的帖子会被同批的大 V 挤掉」

**流行说法**:和大号同时段发会被压,要错开大 V 的发帖时间。

**翻开源码**:排序用的那个 transformer,有一个设计叫**候选隔离注意力掩码**(`phoenix/grok.py:39-71`)。

它的做法:先给整条输入序列建一个因果掩码(`jnp.tril`),然后把"候选区 → 候选区"这一整块注意力**置 0**,只在对角线上,放开每条候选"看自己"。

效果就是:**每条候选,只能看到用户的上下文和它自己,看不见同一批里的其他候选。**

README 把这条单列为 Key Design Decision #2,原话:一条帖子的分数"doesn't depend on which other posts are in the batch"(`README.md:327-328`)。

**所以**:你的帖子得几分,只取决于 **(你的帖子 × 这个用户)** 这一对组合。同批里有没有大 V、大 V 多强,**完全不影响你的分数**。

这是"单独判卷",不是"同场打擂"。打分阶段不存在"被别人挤掉分数"这回事。

(真正的高低比拼发生在打分**之后**的选择阶段——所有候选按分数排序、取前 K 名。那是结果的排名,不是过程里的互相干扰。错开大 V 发帖,救不了一条分数本来就不够高的帖子。)

## 迷思六:「有一套人人适用的涨号攻略」

**流行说法**:跟着某套通用攻略做,就能涨。

**翻开源码**:两个事实。

第一,模型的核心输入是**这一个用户的行为历史**,打分是 per-(用户, 候选) 的预测(`README.md:55, 230`)。同一条内容,对行为历史不同的两个人,分数不一样。

第二,所有的权重——22 个行为权重、多样性衰减的 `decay`、OON 系数——**全部来自可配置参数**(`ranking_scorer.rs:42-66`)。X 不需要发版,就能随时调,还能拿来跑 A/B 实验。系统里甚至还挂着一个可选的二次重排器 `VMRanker`,由一个开关参数控制(`vm_ranker.rs:18-20`)。

**所以**:不存在一张固定的、人人通用的"权重表"。

同一条内容对不同用户分数不同;今天和下个月的权重也可能不同。任何宣称"算法就吃这一套"的通用攻略,面对一个 **per-user、参数化、还在持续 A/B** 的系统,本质上是**刻舟求剑**。

## 那些真正在起作用、却几乎没人提的机制

把上面六条反过来看,你会发现:大多数解读连这套系统**真正的关键词**都没说出口。列几个:

- **候选隔离注意力掩码**——上面迷思五讲的那个,是这套排序模型最精巧的设计之一。
- **双塔召回(Two-Tower)**——你之所以会刷到陌生人,是因为有一个"用户塔"把你压成一个向量,一个"候选塔"把每条帖子压成向量,再比相似度取 top-K(`README.md:191-195`)。这才是"破圈"的真实技术路径。
- **哈希嵌入(Hash-Based Embeddings)**——用户和帖子数量太大,系统用多个哈希函数,把 ID 压进固定大小的嵌入表(`README.md:330-331`)。
- **`not_dwelled` 负信号**——"划走没停留"被单独建模成一个有负权重的预测目标。运营圈天天说"完播率""停留",却很少有人知道它的反面在源码里是一个独立扣分项。
- **Grox 后台内容理解**——在你打开 App 之前,有一套独立服务已经在给新帖子贴标签:是不是 spam、安不安全、质量高不高(内部叫 banger 分)(`README.md:34`)。
- **"删光人工特征"本身**——一个工业级推荐系统,打分侧不靠任何人工特征,全交给一个 transformer 从行为序列里学。这件事本身,就是整个开源仓库里最反常识的一点。

这些才是该写进解读里的东西。

## 怎么一眼识破"AI 废话解读"

你不需要自己读源码,也能筛掉大部分垃圾分析。三个判据:

1. **它敢不敢给"文件:行号"。** 给不出来,基本可以断定:没翻过源码,只是在转述别的解读,而别的解读也没翻过。
2. **它分不分得清"打分"和"过滤"。** 打分靠模型学习,过滤靠明确规则——这是两套完全不同的东西。把它们混着讲的,没读懂。
3. **它承不承认边界。** 一篇斩钉截铁告诉你"权重就是 X""做这个能涨 30%"的文章,在编——因为那些数值根本不在开源代码里。

---

## 最后:一篇诚实的解读,该说清楚的边界

为了不变成我自己批评的那种人,这里把话说全。

**这个开源仓库给了什么**:算法的机制代码(Rust 写的在线服务 + Python 写的 ML 部分)、一个用于演示的 mini 模型(256 维嵌入、4 个注意力头、2 层 transformer,`README.md:32`)、一条能跑通的端到端推理脚本。

**它没有给什么**:线上真实的参数数值(所有权重和系数)、生产级的训练模型、训练数据、Grok 的完整能力。

所以,本文所有结论都是**机制层面**的——"方向是降权""这是一个负权重""分数互不影响"。它们写死在代码里,可核对。但凡涉及**具体数值**的——"降到几折""涨多少个点"——我不会给,因为没人能从这个仓库里得到。

承认这条边界,恰恰就是和那 95% 的区别。真正有价值的解读,不是把话说得最满的那篇,而是**把每句话都钉在能核对的地方**的那篇。

你看到的每一行 `文件:行号`,都欢迎你自己去翻。
